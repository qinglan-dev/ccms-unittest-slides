📖 没有测试的代价

上周上线一个维修记录列表的分页修改，改完我测了新增、没测查询。上线后发现查询按钮点了没反应 —— 原来 getAllByPage 的参数顺序改了，Mapper 返回了空数据。回滚花了一小时。

如果有这个测试 👇 只要 3 秒就能发现问题：

@Test
public void getAllWorkOrders_shouldReturnList() {
    when(mapper.getAll(any())).thenReturn(mockList);
    Result result = service.getAllWorkOrders(null);
    assertNotNull(result.getData());
}

测试是防护网 —— 以后重构代码时，有测试罩着才敢改。

🧩 核心概念：单元测试 & Mock

┌─────────────────────────┐
│   你要测的代码            │
│   addWorkOrder()         │
│                          │
│   1. UUID 生成           │
│   2. Date now            │
│   3. deleteSign = 0      │
│   4. status = "0"        │
│   5. workOrderMapper     │
│      .insert()  ← 依赖   │
└─────────────────────────┘
单元测试 = 不启动数据库、
         不启动服务器的情况下，
         验证代码逻辑是否正确。

真实数据库  ←→  真的开车上路
Mock Mapper ←→  驾校模拟驾驶器

三个注解的口诀

@Mock → "做个假的" @InjectMocks → "装上去" @Captor → "接住"

口诀：先调用 → 再捕获 → 后断言

🔍 被测代码：addWorkOrder 7 步

学生手写的 addWorkOrder 方法：

public Result addWorkOrder(CcmsSmartleasingWorkOrder workOrder, String token) {
    // 1. UUID 主键
    workOrder.setPkWorkOrderId(UUID.randomUUID().toString());
    // 2-3. 审计时间
    Date now = new Date();
    workOrder.setCreationDate(now);
    workOrder.setLastUpdateDate(now);
    // 4. 软删除标记
    workOrder.setDeleteSign((byte) 0);
    // 5. 状态默认 "0=待处理"
    workOrder.setWorkOrderStatus("0");
    // 6. 默认 createdBy / lastUpdatedBy = 0
    if (workOrder.getCreatedBy() == null) workOrder.setCreatedBy(0);
    if (workOrder.getLastUpdatedBy() == null) workOrder.setLastUpdatedBy(0);
    // 7. 调用 Mapper
    workOrderMapper.insert(workOrder);
    return Result.success();
}

7 步就是 7 条业务规则，每条都要写断言来确认。

💻 演示：新建测试类

定位到测试目录：

ccms-smartleasing-impl/src/test/java/com/smartleasing/microservice/
                                                      └── WorkOrderServiceImplTest.java

类签名与注解：

@RunWith(MockitoJUnitRunner.Silent.class)
public class WorkOrderServiceImplTest {

    @Mock
    private CcmsSmartleasingWorkOrderMapper workOrderMapper;

    @InjectMocks
    private WorkOrderServiceImpl workOrderService;

    @Captor
    private ArgumentCaptor<CcmsSmartleasingWorkOrder> workOrderCaptor;
}
  • @Mock → 创建假的 Mapper
  • @InjectMocks → 把假的装进 ServiceImpl
  • @Captor → 准备盒子接住 insert 的参数

✍️ 第一个测试方法（完整）

@Test
public void addWorkOrder_shouldSetAllRequiredFields_whenRequestValid() {
    // ── 准备（3A：Arrange）──
    CcmsSmartleasingWorkOrder request = new CcmsSmartleasingWorkOrder();
    request.setWorkOrderContent("水龙头漏水");
    request.setSubmitter("张三");
    request.setSubmitterContactInformation("13800000000");
    request.setFkWorkOrderTypeId("001");

    // ── 执行（3A：Act）──
    Result result = workOrderService.addWorkOrder(request, null);

    // ── 验证（3A：Assert）──
    // 验证 1：返回值
    assertEquals("code 应为 10000", 10000, result.getCode());

    // 验证 2：verify insert 被调用
    verify(workOrderMapper, times(1)).insert(workOrderCaptor.capture());
    CcmsSmartleasingWorkOrder captured = workOrderCaptor.getValue();

    // 验证 3：逐字段断言 7 步
    assertNotNull("主键不能为空", captured.getPkWorkOrderId());
    assertNotNull("creationDate 必须设置", captured.getCreationDate());
    assertEquals("deleteSign 应为 0", Byte.valueOf((byte)0), captured.getDeleteSign());
    assertEquals("workOrderStatus 应为 '0'", "0", captured.getWorkOrderStatus());
    assertEquals("createdBy 默认应为 0", Integer.valueOf(0), captured.getCreatedBy());

    // 验证 4：业务字段不丢
    assertEquals("工单内容要保留", "水龙头漏水", captured.getWorkOrderContent());
}

🧪 ArgumentCaptor 图解

request (前端传来的)
├─ content: "水龙头漏水"
├─ submitter: "张三"
└─ pkWorkOrderId: null

        ↓ addWorkOrder(request, null)

ServiceImpl 内部做了 7 步修改：
1. 设 UUID → pkWorkOrderId
2. 设 Date → creationDate
3. 设 Date → lastUpdateDate
4. 设 (byte)0 → deleteSign
5. 设 "0"  → workOrderStatus
...

        ↓ insert(workOrder)

captured = captor.getValue()
├─ pkWorkOrderId: "uuid-xxx"  ✅
├─ creationDate: 2026-07-03   ✅
├─ deleteSign: 0              ✅
└─ content: "水龙头漏水"       ✅ 业务字段没丢

关键是：不能用原始 request 做断言 —— 它还没被修改过。

📋 剩余 3 个测试方法

// 测试 2：默认值兜底
@Test
public void addWorkOrder_shouldOverrideNullCreatedBy_withDefaultZero() {
    CcmsSmartleasingWorkOrder request =
        new CcmsSmartleasingWorkOrder();
    workOrderService.addWorkOrder(request, null);
    verify(workOrderMapper).insert(workOrderCaptor.capture());
    assertEquals(Integer.valueOf(0),
        workOrderCaptor.getValue().getCreatedBy());
    assertEquals(Integer.valueOf(0),
        workOrderCaptor.getValue().getLastUpdatedBy());
}
// 测试 3：自定义值保留
@Test
public void addWorkOrder_shouldPreserveExplicitCreatedBy_whenProvided() {
    CcmsSmartleasingWorkOrder request =
        new CcmsSmartleasingWorkOrder();
    request.setCreatedBy(42);
    workOrderService.addWorkOrder(request, null);
    verify(workOrderMapper).insert(workOrderCaptor.capture());
    assertEquals(Integer.valueOf(42),
        workOrderCaptor.getValue().getCreatedBy());
}

// 测试 4：UUID 唯一性
@Test
public void addWorkOrder_shouldGenerateDifferentIdOnEachCall() {
    CcmsSmartleasingWorkOrder first = new CcmsSmartleasingWorkOrder();
    workOrderService.addWorkOrder(first, null);
    CcmsSmartleasingWorkOrder second = new CcmsSmartleasingWorkOrder();
    workOrderService.addWorkOrder(second, null);
    assertFalse("每次应生成不同主键",
        first.getPkWorkOrderId().equals(second.getPkWorkOrderId()));
}

4 个测试，分别验证：成功场景默认值兜底自定义值保留幂等性

🎮 Controller 测试

Controller 不做业务逻辑，只负责"把请求转给 Service，把结果返回给前端"。
@RunWith(MockitoJUnitRunner.class)
public class WorkOrderControllerTest {

    @Mock
    private WorkOrderService workOrderService;

    @InjectMocks
    private WorkOrderController workOrderController;

    @Test
    public void addWorkOrder_shouldDelegateToService() {
        CcmsSmartleasingWorkOrder request =
            new CcmsSmartleasingWorkOrder();
        when(workOrderService.addWorkOrder(request, null))
            .thenReturn(Result.success());

        Result result = workOrderController
            .addWorkOrder(request, null);

        assertEquals(10000, result.getCode());
        verify(workOrderService)
            .addWorkOrder(request, null);
    }
}
Service 测试Controller 测试
Mock MapperMock Service
验证 insert 参数验证方法被调用
7 个断言2 个断言

Controller 测试比 Service 测试简单得多 —— 因为业务逻辑不在 Controller 里。

🔴 读懂报错 —— 测试的价值在红色时

故意改断言，投屏展示红色报错：

java.lang.AssertionError: code 预期失败
Expected: 9999
Actual: 10000

读报错三要素：

  1. Expected —— 你期望的值（9999）
  2. Actual —— 实际值（10000）
  3. 哪个文件哪一行 —— 定位到出错的断言
"第 6 步很重要 —— '看得懂红色'才算学会了。"

⚠️ 常见报错 & 排查

No tests found

原因：测试类名不是 *Test.java，或放错了目录

WorkOrderServiceImplTest.java  ✅
WorkOrderTestServiceImpl.java  ❌
WorkOrderServiceImplTestCases.java  ❌

Wanted but not invoked

Wanted but not invoked:
workOrderMapper.insert(...);
Actually, there were zero interactions with this mock.

说明 ServiceImpl 里没有调 workOrderMapper.insert()。

Expected vs got

Expected: is "0"
     got: null

expected '0' 但 got null → setWorkOrderStatus("0") 没执行。

🎬 总结

1️⃣ 单元测试不是黑魔法

它就是写几个 assert 验证你的代码。

2️⃣ Mock 是隔离术

用假的替代真的，让测试只测你的代码。

3️⃣ 测试要可进化

以后加功能时，测试跟着加，不是交完作业就丢。

        if (workOrder.getLastUpdatedBy() == null) workOrder.setLastUpdatedBy(0);
        workOrderMapper.insert(workOrder);
        return Result.success();
    }
}
